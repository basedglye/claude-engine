# Asset import pipeline fixtures

These fixtures are the mechanically-checked exit criterion for
docs/PHASE-2.md Scope C (exit criterion 6): the good fixtures must pass
`validateAsset`/`assets:import` (exit 0), and the bad fixtures must each fail
with the specific gate named in `gates[]` (exit 1).

All fixtures were generated by hand (a one-off Node script using only
built-in `node:zlib`/`node:fs` — no PNG/glTF library), matching the
constraint the shipped validator itself follows: no image- or
mesh-decoding dependency in `@claude-engine/asset-pipeline`.

## `good-texture.png`

A real, decodable 4x4 RGB PNG (power-of-two dimensions, ~70 bytes). Exercises
the happy path for `type: "texture"`:

- `format` — valid PNG magic bytes: pass
- `size` — well under `DEFAULT_BUDGETS.texture.maxBytes` (2 MiB): pass
- `dimensions` — 4x4 ≤ `maxDim` (2048): pass
- `pow2` — 4x4 is power-of-two, and `requirePow2` is `true`: pass

## `bad-texture-oversized.png`

A structurally valid PNG (correct magic bytes, IHDR chunk, IDAT/IEND with
correct CRCs) whose IHDR **claims** 4096x4096 — over
`DEFAULT_BUDGETS.texture.maxDim` (2048). The actual encoded pixel data is a
tiny 1x1 image (the validator only reads the IHDR header fields for the
dimensions gate, never decodes IDAT, so this keeps the fixture small while
exercising exactly one gate):

- `format`: pass (still a valid PNG container)
- `size`: pass (file is tiny)
- `dimensions`: **fail** — `4096x4096` exceeds `maxDim` 2048
- `pow2`: pass (4096 is incidentally power-of-two — this fixture is
  specifically about the dimensions budget, not pow2)

## `good-mesh.glb`

A minimal, structurally valid glTF binary: one triangle (3 positions, 3
indices, `mode: 4` / TRIANGLES), zero materials. Exercises the happy path for
`type: "mesh"`:

- `format` — valid `glTF` binary container with a readable JSON chunk: pass
- `size` — ~628 bytes, well under `maxBytes` (5 MiB): pass
- `triCount` — 1 triangle ≤ `maxTris` (20,000): pass
- `materialCount` — 0 materials ≤ `maxMaterials` (8): pass

## `bad-mesh-overtris.glb`

Structurally identical to `good-mesh.glb` (same tiny backing buffer), except
the indices accessor's `count` metadata is set to `300000` (⇒ 100,000
claimed triangles), which exceeds `DEFAULT_BUDGETS.mesh.maxTris` (20,000).
Per docs/PHASE-2.md Scope C, the validator trusts accessor `count` metadata
directly rather than cross-checking it against actual buffer byte lengths,
so a fixture is free to hand-construct claimed metadata like this without a
buffer that would actually hold 300,000 indices:

- `format`: pass (still a structurally valid `.glb`)
- `size`: pass (file is tiny — the buffer data is fake, not the real size a
  100k-triangle mesh would need)
- `triCount`: **fail** — 100,000 claimed triangles exceeds `maxTris` 20,000
- `materialCount`: pass (0 materials)

## Regenerating

No generator script is committed (these files are static, deterministic
fixtures and are not expected to change). If they ever need to be
regenerated by hand, `good-texture.png`/`bad-texture-oversized.png` are
built by hand-assembling a PNG signature + `IHDR`/`IDAT`/`IEND` chunks (CRCs
via `node:zlib`'s `crc32`, pixel data compressed via `deflateSync`);
`good-mesh.glb`/`bad-mesh-overtris.glb` are built by hand-assembling the
12-byte glTF binary header followed by a `JSON` chunk (padded to a 4-byte
boundary with spaces) and a `BIN\0` chunk (padded with zero bytes).
